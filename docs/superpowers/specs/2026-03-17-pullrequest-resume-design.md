# PR Resume — Design Spec

## Context

Developers need visibility into what was accomplished each day across a repository. Currently, this requires manually reviewing GitHub PRs and commits. This project automates that by pulling PR/commit data from GitHub and using OpenAI GPT to generate a natural language summary of the day's work.

## Overview

A **Next.js 14+ (App Router)** full-stack application that:

1. Connects to the GitHub API to fetch PRs and commits for a specific repository and date
2. Sends the collected data to OpenAI GPT to generate a human-readable summary
3. Displays the summary and raw data in a web dashboard

## Tech Stack

- **Framework**: Next.js 14+ (App Router, Server Components)
- **Language**: TypeScript
- **Styling**: Tailwind CSS
- **GitHub Integration**: Octokit (official GitHub SDK)
- **AI**: OpenAI SDK (GPT)
- **Config**: Environment variables (`.env.local`)

## Architecture

```
Browser (Dashboard)
    │
    ▼
Next.js App Router
    ├── /                  → Main page (today's summary)
    ├── /config            → Repository configuration
    └── /api/
        ├── summary        → GET: fetch data + generate summary
        └── config         → GET/POST: read/save repo config
            │
            ├── GitHub API (Octokit) → PRs, Commits
            └── OpenAI API → Natural language summary
```

## Config Storage

Repository configuration (owner, repo) is stored in environment variables (`.env.local`). The `/config` page reads from env vars and does NOT write to them at runtime — it only validates that the connection works. To change the repo, the user edits `.env.local` and restarts the server. This keeps v1 simple and avoids file-write complexity.

## Data Flow

1. User opens dashboard (default: today's date, based on server's UTC timezone)
2. Frontend calls `GET /api/summary?date=YYYY-MM-DD`
3. API Route fetches from GitHub:
   - PRs merged on the given date (default branch only, excludes drafts)
   - Commits on the default branch for the given date (using `committer-date`)
   - Commits that are part of listed PRs are deduplicated (shown only under the PR)
   - For each PR: title, description, author (file changes and reviews are omitted to reduce API calls)
4. API Route builds a prompt with the collected data
5. Prompt is sent to OpenAI GPT (`gpt-4o-mini` — fast, cheap, sufficient for summarization)
6. API returns JSON response to frontend
7. Frontend renders summary + structured data

### GitHub API Rate Limits

- Authenticated requests: 5,000/hour. A typical summary request makes ~3-5 API calls (list PRs, list commits, PR details)
- If rate limited (HTTP 403 with `X-RateLimit-Remaining: 0`), the API returns a 429 error with the reset time
- No automatic retry — the user can try again after the reset window

### OpenAI Cost Controls

- Model: `gpt-4o-mini` (low cost per token)
- Max input: if collected data exceeds 4,000 tokens, truncate to PR titles + commit messages only (drop descriptions)
- API timeout: 30 seconds
- Max output tokens: 500

## API Design

### `GET /api/summary`

**Query Parameters**:
| Param  | Type   | Required | Default | Description              |
|--------|--------|----------|---------|--------------------------|
| date   | string | No       | today   | Date in YYYY-MM-DD format |

Repository owner/repo are read from server-side environment config.

**Response** (200):
```json
{
  "date": "2026-03-17",
  "repository": "owner/repo",
  "summary": "Natural language summary generated by GPT...",
  "pullRequests": [
    {
      "number": 42,
      "title": "Add user authentication",
      "author": "username",
      "status": "merged",
      "url": "https://github.com/..."
    }
  ],
  "commits": [
    {
      "sha": "abc1234",
      "message": "feat: add login page",
      "author": "username",
      "date": "2026-03-17T14:30:00Z"
    }
  ],
  "stats": {
    "totalPRs": 3,
    "totalCommits": 12,
    "contributors": ["user1", "user2"]
  }
}
```

**Error responses**:
```json
{
  "error": {
    "code": "INVALID_DATE | MISSING_CONFIG | GITHUB_AUTH_ERROR | OPENAI_ERROR | RATE_LIMITED | INTERNAL_ERROR",
    "message": "Human-readable error description",
    "retryAfter": 1234567890
  }
}
```
- 400: invalid date format
- 401: bad GitHub or OpenAI token
- 429: GitHub rate limit hit (includes `retryAfter` timestamp)
- 500: unexpected server error

### `GET /api/config`

Returns current repository configuration (owner, repo from env vars) and connection status (validates token + repo access).

## Frontend Pages

### `/` — Dashboard (Main Page)

- **Header**: Repository name, link to config
- **Date picker**: Navigate between dates (defaults to today)
- **Summary card**: GPT-generated summary rendered as Markdown
- **Pull Requests section**: Table/list with number, title, author, status, link
- **Commits section**: Table/list with short SHA, message, author, timestamp
- **Stats bar**: Total PRs, total commits, unique contributors
- **Loading state**: Skeleton UI while fetching
- **Empty state**: Message when no activity for the selected date

### `/config` — Configuration Status

- **Shows**: Current owner/repo from environment variables
- **Validate button**: Tests GitHub API connection and OpenAI key
- **Status indicators**: Green/red for each service
- **Instructions**: How to change config via `.env.local`

## GPT Prompt Strategy

The prompt will:
1. Receive structured data about PRs and commits
2. Be instructed to generate a concise daily summary
3. Group related changes by feature/area when possible
4. Highlight significant changes (large PRs, breaking changes)
5. Mention contributors
6. Keep the tone professional but approachable

Example system prompt:
```
You are a development team assistant. Given a list of pull requests
and commits from a GitHub repository for a specific date, generate a
concise summary of what was accomplished. Include the date in your
summary. Group related changes by feature or area. Highlight
significant changes. Mention contributors. Keep it under 300 words.
```

The user message includes the date and the structured data.

## Testing Strategy

- **Manual testing**: Run dev server, configure a real repo, verify summary generation
- **API testing**: Test `/api/summary` with different dates and edge cases (no activity, many PRs)
- **Edge cases**: Invalid dates, missing tokens, API rate limits, repos with no activity

## Out of Scope (for v1)

- Database/persistence for historical summaries
- Authentication/multi-user support
- Multiple repository support
- Scheduled/automatic summary generation
- Slack/Discord integrations
- Caching layer
- Pagination for very active repos (100+ PRs/day)
